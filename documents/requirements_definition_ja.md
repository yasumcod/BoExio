# BoExio 要件定義書

作成日: 2026-05-21

参照元: `documents/product_vision_ja.md`

## 1. 本書の目的

本書は、BoExio の初期開発と段階的な運用拡張に必要な業務要件、機能要件、データ要件、非機能要件、受け入れ条件を定義する。

BoExio は、BoConcept の公開商品ページから商品情報・構成情報・価格情報を継続的に取得し、商品マスタ、価格差分、営業向けレポートを生成する業務支援ツールである。

## 2. 背景と課題

現在の商品情報確認と見積作成は、人の目視確認と転記に依存している。そのため、価格改定の見落とし、商品構成ごとの価格差確認漏れ、Excel や見積書への転記ミス、変更履歴の欠落が発生しやすい。

BoExio は、商品マスタの自動生成と価格差分の自動検知により、営業担当者と管理者が最新価格・新商品・販売終了候補・取得失敗を確認できる状態を作る。

## 3. プロダクト概要

### 3.1 プロダクト名

BoExio

### 3.2 一言説明

BoConcept の商品価格と構成情報を自動取得し、価格変更を検知して営業見積に使える商品マスタと差分レポートを生成する業務支援ツール。

### 3.3 主要利用者

- 営業担当: 最新価格を確認し、見積ミスを減らす
- 営業管理者: 価格改定、新商品、販売終了候補を把握する
- 運用担当者: 商品マスタ更新作業を手作業から外す
- 将来の見積システム: 正規化された商品・価格データを利用する

## 4. スコープ

### 4.1 MVP スコープ

MVP は、全自動の完成品ではなく、価格差分検知までの最短ルートを作る。

- 対象ドメインは `https://www.boconcept.com` とする
- 実取得対象は `https://www.boconcept.com/ja-jp/shop/` 配下に限定する
- 初期対象カテゴリはチェアとする
- 初期対象 URL は `https://www.boconcept.com/ja-jp/shop/%E3%83%81%E3%82%A7%E3%82%A2/` とする
- 対象 URL リストは手動管理とする
- 第一段階ではチェアカテゴリページ 1 件を取得対象にする
- 取得できた内容を生データ CSV として出力する
- 商品構成ごとに 1 行のデータへ正規化する
- 前回 CSV と今回 CSV を比較し、価格変更、新規追加、削除候補を出力する
- 後続段階で営業が確認できる Excel レポートを生成する

### 4.2 MVP で扱わない範囲

- フル機能の Web 管理画面
- 見積書の完全自動作成
- 大量商品を前提にしたクラウド API 化
- 社内顧客データとの連携
- 複雑な承認ワークフロー

### 4.3 将来スコープ

- 複数カテゴリへの対象拡大
- 全商品カテゴリへの対象拡大
- GitHub Actions による週次自動実行
- GitHub Releases への成果物保存
- 見積運用との連携
- 必要に応じた管理画面、API、クラウド実行環境の追加

## 5. 業務要件

### 5.1 営業担当向け要件

- 毎週生成される成果物を確認すれば、価格変更、新商品、削除候補、取得失敗を把握できること
- 見積作成時に最新の商品マスタを参照できること
- 値上がり、値下がり、見積影響のある変更を確認できること
- 商品 URL から元ページを確認できること

### 5.2 営業管理者向け要件

- 価格改定履歴を実行日単位で確認できること
- 過去の見積や価格監査に利用できる履歴が残ること
- 毎週の成果物を GitHub Releases から取得できること
- 変更件数、取得成功数、取得失敗数を俯瞰できること

### 5.3 運用担当者向け要件

- 対象 URL を追加すれば、定期実行で商品マスタと差分レポートを生成できること
- 取得失敗、解析失敗、比較キー生成失敗がログとエラー一覧に残ること
- エラー内容から修正対象 URL と処理フェーズを特定できること
- 実行結果が失敗した場合でも、生成済み成果物とエラー情報を保存できること

## 6. 機能要件

### 6.1 対象 URL 管理

- システムは、手動管理された対象 URL リストを読み込めること
- システムは、`https://www.boconcept.com/ja-jp/shop/` 配下以外の URL を除外できること
- システムは、robots.txt の Disallow パターンに一致する URL を投入時と実行時の 2 段階でスキップできること
- 初期 URL リストにはチェアカテゴリ URL を含めること

### 6.2 商品情報取得

- システムは、対象 URL から商品ページまたは商品一覧に関する情報を取得できること
- 取得戦略は、内部 JSON/API、HTML 内の構造化データ、Playwright による UI 操作、URL パターン推測の順に検討すること
- システムは、商品名、商品番号、商品 URL、価格、構成情報、寸法、素材、画像 URL、PDF URL などを取得対象として扱えること
- 取得できない項目は空欄とし、推測による補完を避けること
- 取得失敗 URL も失敗行として CSV に残せること

### 6.3 構成単位の正規化

- システムは、商品単位ではなく構成単位で 1 レコードを生成すること
- 構成は、サイズ、張地、脚、SKU、構成 ID などの取得可能な属性で表現すること
- 構成ごとに価格や SKU が異なる場合、構成ごとの行として出力すること
- 構成単位の比較キーを生成できない行は、通常の差分比較から除外し、errors に出力すること

### 6.4 CSV 出力

- システムは、今回取得分の商品マスタ CSV を出力できること
- システムは、日付付きスナップショット CSV を保存できること
- 生データ CSV には成功行と失敗行を含めること
- 成功行は `scrape_status=success`、失敗行は `scrape_status=failed` とすること
- 各行には `source_url` と `source_checked_at` を含めること

### 6.5 差分検知

- システムは、前回 CSV と今回 CSV を比較できること
- システムは、価格変更、新規追加、削除候補を検知できること
- 価格が欠損している場合、価格ソースが一致しない場合、税区分や通貨が一致しない場合は、価格変更として扱わず errors に記録すること
- 比較キーが生成できないレコードは、added、removed、price_changes の判定対象から除外すること
- 比較ロジックにはテストを用意すること

### 6.6 削除候補と販売終了状態管理

- システムは、removed 判定を即時販売終了とは扱わず、削除候補として扱うこと
- 状態は `active`、`missing_candidate`、`discontinued`、`revived` の 4 種類とすること
- 週次実行を前提に、連続 4 回未検知で `discontinued` とすること
- `discontinued` 済みの比較キーが再検知された場合は `revived` とすること
- revived の場合は、復活検知日、直前の discontinued 判定日、連続未検知回数、復活後初回価格を記録すること

### 6.7 Excel レポート生成

- 後続段階で、営業が確認しやすい Excel レポートを生成できること
- Excel レポートは `summary`、`price_changes`、`added`、`removed`、`current_master`、`errors` のシートを含むこと
- 金額、差額、変化率は確認しやすい形式で整形すること
- errors シートは、取得失敗や解析失敗を運用者が確認できる形式にすること

### 6.8 実行履歴と成果物保存

- システムは、各実行を `run_id` で識別できること
- システムは、CSV、Excel、run metadata、ログを成果物として保存できること
- GitHub Actions 運用時は、成果物を GitHub Releases の assets として保存できること
- 同日複数実行時でも、`run_id` により成果物とメタデータを 1 対 1 で追跡できること

### 6.9 定期実行

- GitHub Actions により週次 cron 実行できること
- 手動実行もできること
- 実行基準時刻とタイムゾーンは運用開始前に確定すること
- Private Repository で運用できること
- 失敗時にログと errors を確認できること

## 8. 比較キー要件

比較キーは、取得可否に応じて次の優先順位で生成する。

1. `variant_id`
2. `sku`
3. 正規化済み属性連結キー: `item_number + selected_size + selected_upholstery + selected_leg`

属性連結キーを使う場合、次の正規化を必須とする。

- 前後空白を除去する
- 英字は小文字化して比較する
- 英数字、記号、スペースを NFKC で正規化する
- 比較に不要な区切り記号を統一ルールで除去または単一記号に寄せる
- 連続空白を 1 つに圧縮し、比較前に不要空白を除去する
- 代表的な同義表現を表記ゆれ辞書で正規化する

比較キー生成失敗時は、`missing_required_attribute`、`conflicting_identifiers`、`normalization_failed`、`empty_key_after_normalization` などのエラー分類で errors に出力する。

比較キー方式を変更する場合は、一定期間は旧キーと新キーを併記し、履歴整合性を確認できるようにする。切替日と採用方針は運用ドキュメントに記録する。

## 9. 価格差分判定要件

- MVP では、取得できた価格文字列または数値を過度に整形せず、同一ソース同士で比較する
- 採用した価格ソースは `price_from` に記録する
- 価格が欠損している場合は、価格変更として扱わず errors に記録する
- 前回と今回で価格ソースが一致しない場合は、価格変更として扱わず errors に記録する
- 税込/税抜の自動換算は MVP では行わない
- 差分比較は同じ税区分同士に限定する
- 差分比較は同一通貨間に限定する
- 異通貨は自動換算せず、`currency_mismatch` 警告として errors と summary に件数を出す
- `canonical_price` を使った厳密な正規化は、取得実態を確認した後の後続フェーズで設計する

## 10. 出力要件

### 10.1 CSV

初期段階で生成する主な成果物は次の通り。

- `products_current.csv`
- `products_YYYY-MM-DD.csv`
- `price_changes_YYYY-MM-DD.csv`
- `new_items_YYYY-MM-DD.csv`
- `removed_items_YYYY-MM-DD.csv`
- `scrape_log_YYYY-MM-DD.txt`

同日複数実行が発生する場合は、メタデータで `run_id` と 1 対 1 対応させるか、`products_YYYY-MM-DD_<run_id>.csv` のように `run_id` をファイル名に含める。

### 10.2 Excel レポート

後続段階で `weekly_report_YYYY-MM-DD.xlsx` を生成する。

必須シート:

- `summary`
- `price_changes`
- `added`
- `removed`
- `current_master`
- `errors`

### 10.3 summary シート

summary には次の指標を含める。

- 取得日
- 対象商品数
- 取得成功数
- 取得失敗数
- 総構成数
- 価格変更数
- 値上げ数
- 値下げ数
- 新規追加数
- 削除候補数
- 新規候補数
- 確定終了数
- 復活数
- 通貨不一致件数
- 比較不可件数

### 10.4 price_changes シート

price_changes には次の項目を含める。

- 商品名
- 商品番号
- サイズ
- 張地
- 脚
- 前回価格
- 今回価格
- 差額
- 変化率
- 商品 URL
- 取得日時

### 10.5 removed シート

removed には削除候補、販売終了確定、復活を確認できる情報を含める。

- `variant_key`
- 商品名
- 商品番号
- 現在状態
- 連続未検知回数
- 初回未検知日
- discontinued 判定日
- revived 検知日
- 復活後初回価格
- 商品 URL

### 10.6 errors シート

errors シートの必須列は次の通り。

- `url`
- `phase`
- `error_code`
- `message`
- `first_seen_at`
- `last_seen_at`

`phase` は `fetch`、`render`、`parse`、`normalize`、`diff`、`report` のいずれかとする。必須列が欠損している場合、レポート生成は失敗扱いにする。

## 11. エラー要件

エラーコード体系は次を標準とする。

| エラーコード | 意味 |
| --- | --- |
| `HTTP_<status>` | HTTP ステータス起因 |
| `TIMEOUT_CONNECT` | 接続タイムアウト |
| `TIMEOUT_READ` | 応答タイムアウト |
| `TIMEOUT_RENDER` | 描画待ちタイムアウト |
| `SELECTOR_MISS` | 期待した DOM セレクタ要素が取得不可 |
| `SCHEMA_MISMATCH` | JSON/HTML 構造が想定スキーマと不一致 |
| `PARSE_ERROR` | 価格、SKU、寸法などの値変換失敗 |
| `RATE_LIMITED` | サイト側レート制限を検知 |
| `UNKNOWN` | 上記に分類できない例外 |

`UNKNOWN` の場合も、message に原因調査に必要な生ログ要約を残す。

## 12. 実行制御要件

### 12.1 URL 単位の再試行

- `max_retries_per_url = 3` とする
- 初回失敗後に最大 3 回再試行し、合計 4 試行までとする
- backoff は指数方式とジッターを組み合わせる
- `HTTP_404` と `SCHEMA_MISMATCH` は非再試行とする
- `HTTP_429`、`HTTP_5xx`、`TIMEOUT_*`、`RATE_LIMITED` は再試行対象とする

### 12.2 URL 単位の打ち切り

- 同一 URL で再試行上限に到達した場合は打ち切る
- 単一 URL の累積処理時間が `max_url_runtime_sec` を超えた場合は打ち切る
- `SELECTOR_MISS` または `PARSE_ERROR` が連続 2 回発生した場合は DOM 変更疑いとして打ち切る

### 12.3 実行全体の失敗判定

- `failure_rate = failure_count / target_count` とする
- `failure_rate > 0.30` の場合は `run_status=failed` とする
- `target_count >= 20` かつ `failure_count >= 5` の場合も `run_status=failed` とする
- `SCHEMA_MISMATCH` が全体で 3 件以上の場合も `run_status=failed` とする
- `run_status=failed` の場合も生成済み成果物と errors シートを保存する

## 13. 非機能要件

### 13.1 正確性

- 価格、SKU、構成名、取得日時が正確であること
- 曖昧なデータは空欄またはエラーとして扱うこと
- 取得できていない情報を推測で補完しすぎないこと

### 13.2 再現性

- 同じ入力 URL と同じ取得日時のデータから、同じ CSV とレポートを生成できること
- 入力 URL 一覧、取得日時、schema_version、parser_version、commit_sha、run_id、実行ログ、出力ファイルチェックサムを保存すること
- 差分比較時は `schema_version` の一致を必須条件とすること
- `schema_version` が不一致の場合は、差分処理を停止するか、互換変換を実施してから比較するかを明示すること

### 13.3 追跡性

- 各行に `source_url` と `source_checked_at` を持たせること
- 成果物ファイルと run metadata を `run_id` で相互参照できること
- 価格差分の根拠となる前回ファイルと今回ファイルを特定できること

### 13.4 運用安全性

- 公開ページのみを対象とすること
- ログイン情報、Cookie、顧客情報、社内価格表を扱わないこと
- 対象範囲は `https://www.boconcept.com/ja-jp/shop/` 配下に限定すること
- robots.txt と利用規約の確認結果を運用開始前に記録すること

### 13.5 アクセス制御

初期値は次を標準とする。

- requests/sec 上限: 平均 0.2 req/sec
- 同時接続数: 1
- クロール時間帯: 02:00-05:00 JST
- リトライ時は指数バックオフとジッターを適用する

### 13.6 ブロック兆候検知

- 直近 20 リクエスト中の 403 比率が 20% 以上になった場合は即時停止する
- captcha 検知が 1 件でも発生した場合は即時停止する
- 自動停止時は成果物生成を中断し、Actions 失敗とログ出力で運用担当に通知する

### 13.7 User-Agent

- User-Agent は固定で明示すること
- User-Agent にはプロジェクト名、用途、連絡先を含めること
- 例: `BoExioPriceMonitor/1.0 (+contact: boexio-ops@example.com)`
- 連絡先は運用チームの受信可能なメールアドレスとすること
- サイト側要請がある場合は、要請内容を優先して User-Agent 表記と連絡先を調整すること

## 14. コンプライアンス要件

- robots.txt の取得・確認を運用開始前に行うこと
- 利用規約の確認を運用開始前に行うこと
- 確認結果を `documents/compliance_checklist.md` 等に記録し、更新日と確認者を残すこと
- URL 投入時とクロール実行時に robots.txt の除外パターンを確認すること
- サイトマップを参照する場合も、抽出対象は `/ja-jp/shop/` 配下 URL のみに限定すること

初期除外対象は次を標準とする。

- `*/search/?*`
- `*/shop/*_*`
- `*/shop/*?q=*`
- `*/on/demandware*`
- `*/p/*/print/`
- `*/store-lead/*`
- `*/undefined/*`
- `*/v/*`
